## Context

Dungeon Toolkit Web 现有后端使用 Django + DRF，前端使用 React + Zustand。D&D 角色卡功能已归档封存。

现在需要为 DicePP 机器人建立展示和登记平台：
- 运营者需要登录来管理自己的机器人
- 机器人信息需要公开给所有用户查看
- 机器人需要定期检测在线状态

## Goals / Non-Goals

**Goals:**
- 实现用户注册/登录系统（Token 认证）
- 实现机器人广场（公开浏览、搜索）
- 实现机器人登记（CRUD、在线状态检测）
- 实现机器人详情页

**Non-Goals:**
- 配置共享功能（后续迭代）
- 基建服务如 Webhook 中转（后续迭代）
- 机器人运行时数据云端存储
- 与 DicePP 机器人的深度集成

## Decisions

### 1. 认证方案：Token vs Session
- **决定**: 使用 Token 认证（DRF Token）
- **理由**: 前端 SPA 应用，Token 更适合；现有 users app 已有基础

### 2. 在线状态检测：主动 vs 被动
- **决定**: 运营者提供健康检查端点，网站定期 HEAD 请求检测
- **理由**: 运营者部署环境多样，主动检测更灵活；机器人本身有 HTTP 服务

### 3. 前端路由设计
- **决定**: 机器人页面使用 `/robots` 前缀，与现有页面分离
- **理由**: 功能独立，避免与角色卡页面冲突

### 4. 数据存储
- **决定**: 使用 Django ORM，存储在现有数据库
- **理由**: 简化部署，与现有系统一致

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 运营者不会配置健康检查端点 | 提供 Docker 部署模板，内置健康检查 |
| 机器人部署环境无法被外网访问 | 暂时仅展示登记信息，基建服务后续实现 |
| 大量机器人检测请求 | 限制检测频率（5分钟），异常时降级 |
